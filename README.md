# Проект: Контейнеризация на проект с Docker Compose

Тази техническа документация описва детайлно процеса по контейнеризация, мрежова изолация и управление на двукомпонентно уеб приложение. Архитектурата интегрира **Node.js/Express REST API** и **MongoDB СУБД**, управлявани по декларативен път чрез **Docker Compose**.

---

##  Каква е структурата на проекта.

Проектът е организиран в следната чиста и изолирана файлова структура:
*   `server.js` — Основният изходен код на уеб сървъра (Бекенд архитектура и бизнес логика).
*   `package.json` — Конфигурационен файл с метаданни, скриптове и зависимости на Node.js проекта.
*   `Dockerfile` — Спецификация и инструкции за изграждане на Docker образа на нашето уеб приложение.
*   `compose.yml` — Централен оркестрационен манифест, който дефинира, свързва и стартира двете основни услуги едновременно.
*   `.gitignore` — Файл, указващ на Git кои директории (напр. тежката папка `node_modules`) да не се споделят в GitHub.

---

##  Как работи всеки от компонентите.

Проектът е разделен на две напълно изолирани микроуслуги, работещи в собствени жизнени цикли:

### 1. Бекенд услуга (`web-app`)
*   **Стек:** Node.js 18 / Express.js / Mongoose ODM.
*   **Роля:** Приема клиентски HTTP заявки, рендерира начална страница с графичен интерфейс и комуникира с базата данни за извършване на CRUD операции (запис и четене на съобщения).
*   **Контейнеризация:** Използва олекотен базов образ `node:18-alpine` с цел оптимизация на паметта и сигурност. Контейнерният порт `3000` е пренасочен към хост порт `3000` (`-p 3000:3000`).

### 2. База данни (`database`)
*   **Стек:** MongoDB Server 6.0.
*   **Роля:** Служи като нерелационно (NoSQL) дълготрайно хранилище за данни.
*   **Контейнеризация:** Работи на стандартния си порт `27017` в изолирана среда. Внедрен е именуван дял (**Named Volume**) с идентификатор `mongo-data`, монтиран към вътрешната директория `/data/db`. Това подсигурява персистентност на данните – записите остават непокътнати на твърдия диск дори след изтриване на контейнера.

---

##  Как е осъществена комуникацията между услугите.

Комуникацията между компонентите е напълно абстрахирана и се осъществява изцяло в изолираната вътрешна мрежа (**Default Bridge Network**), която Docker Compose менажира автоматично:
*   **Service Discovery:** Приложението не разчита на статични IP адреси или `localhost`. Node.js приложението резолва адреса на СУБД чрез DNS името на самата услуга (`database`), дефинирано в `compose.yml`.
*   **ConnectionString:** Експортиран динамично чрез променлива на средата: `MONGO_URI=mongodb://database:27017/my-docker-db`.
*   **Оркестрация на жизнения цикъл:** Използвана е директивата `depends_on`, която налага строга последователност при стартиране. Това гарантира, че базата данни се инициализира преди уеб сървъра, предотвратявайки критични грешки от тип `ECONNREFUSED`.

##  Как се изграждат и стартират контейнерите.

За изграждането и стартирането на инфраструктурата се използва декларативният модел на Docker Compose, който автоматизира целия процес чрез следните стъпки:

1. Уверете се, че софтуерът **Docker Desktop** е стартиран на вашата машина.
2. Отворете CLI терминала директно в кореновата папка на проекта.
3. Изпълнете следната команда, за да стартирате едновременно компилирането на локалния `Dockerfile` и изтеглянето на контейнерните образи:
